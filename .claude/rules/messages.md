# Message Taxonomy Rules (ASTRA Message Registry)

## The Single-Source Principle (Non-Negotiable)

All system messages тАФ UI text, audit log entries, notifications тАФ live in exactly one file: `shared/messages/locales/messages.yaml`. No message string may exist anywhere else. The Redis cache, browser JSON bundles, and `docs/CTS_Msg_Taxonomy.html` are all derived outputs.

---

## Adding a New Message тАФ Mandatory Steps

**STEP 1 тАФ Add to `messages.yaml`** with all required fields:
- `severity:` тАФ INFO | WARN | ERROR | CRITICAL
- `surface:` тАФ any combination of `[UI, AUDIT, NOTIFICATION]`
- `variables:` тАФ list every `{placeholder}` that appears in the text (must match exactly тАФ no extras, no missing)
- `en:` тАФ English text with `{placeholder}` substitutions
- `hi: ""` тАФ empty string stub required; never omit

**Key naming:** `{DOMAIN}_{SUBDOMAIN}_{DESCRIPTOR}` тАФ all uppercase, underscores only.

| Domain prefix | Use for |
|---|---|
| `CTS_WF` | CTS inward cheque processing workflow |
| `CTS_OUT` | CTS outward clearing (scanner, lot, endorsement, NGCH) |
| `CTS_COMP` | CTS-2010 compliance violations |
| `CTS_NGCH` | NGCH filing, IET, transport, settlement |
| `CTS_SMB` | Sub-member bank sponsor routing |
| `CBS` | Core Banking System connector events |
| `VAULT` | Signature vault and PPS vault events |
| `AUTH` | Authentication, session, RBAC, user management |
| `EJ` | ATM Electronic Journal processing and disputes |
| `PLATFORM` | Infrastructure: config, policy, audit, model, DC, Kafka |

**STEP 2 тАФ Run the build:** `python -m shared.messages.build`
This validates YAML, pushes to Redis, writes browser JSON bundles, and regenerates `docs/CTS_Msg_Taxonomy.html`.

**STEP 3 тАФ Commit BOTH files together:**
```
git add shared/messages/locales/messages.yaml docs/CTS_Msg_Taxonomy.html
```
Never commit one without the other. Pre-commit Check 11 enforces this pairing.

## Severity Rules

| Severity | When to use |
|---|---|
| `INFO` | Normal lifecycle events, successful operations (STP confirm, login success, vault hit) |
| `WARN` | Degraded path, soft failures (vault stale, OCR low confidence, CBS unreachable) |
| `ERROR` | Hard failures requiring intervention (NGCH submit failed, signature mismatch) |
| `CRITICAL` | Immediate action required (IET expired, vault stale halting processing, cert expired) |

Rules: vault miss тЖТ WARN minimum ┬╖ IET breach or risk тЖТ CRITICAL ┬╖ account frozen/stop payment тЖТ CRITICAL ┬╖ audit integrity tamper тЖТ CRITICAL ┬╖ CRITICAL messages must include NOTIFICATION surface

---

## Surface Rules

| Surface | Meaning | Rules |
|---|---|---|
| `UI` | Visible to bank operators in the workstation | Include for any event a human reviewer or ops_manager needs to see |
| `AUDIT` | Written to Immudb via AuditEvent | **Mandatory** on any key touching a cheque decision, vault operation, or auth event |
| `NOTIFICATION` | Sent via email / WhatsApp | NOTIFICATION alone (without AUDIT) is forbidden тАФ if worth notifying, it's worth auditing |

---

## Variable Rules

Every `{placeholder}` in `en:` must appear in `variables:` and vice versa. Undeclared or unused variables fail validation (`registry.validate()` returns errors).

Standard variable names: `account_display` (always masked `****4521`) ┬╖ `amount_range` (always bucketed `тВ╣[1L-5L]`) ┬╖ `instrument_id` ┬╖ `bank_id`. Use descriptive names: `return_reason` not `rr`.

---

## Using Messages in Code

Use `from shared.messages import get_message` with keyword args for all placeholders. Use `from shared.messages import get_entry` to access `entry.severity` and `entry.surface` for AuditEvent routing. Never use f-string log messages or hardcoded strings.

---

## `CTS_Msg_Taxonomy.html` тАФ Auto-Generated

Generated by `python -m shared.messages.build`. Do NOT edit directly тАФ it will be overwritten on next build. Always committed alongside `messages.yaml`.

---

## Forbidden
- Hardcoded message strings in any Python, Go, or React file
- `hi:` stub field missing on any key
- `MSG_001` style numeric IDs тАФ descriptive domain-prefixed names only
- Editing `CTS_Msg_Taxonomy.html` directly
- `print()` instead of `structlog` when emitting message content

---

## Enforcement

| Rule | Enforced By | Blocks |
|---|---|---|
| All messages in `messages.yaml` only | Semgrep `astra-no-hardcoded-message-strings` | PR merge blocked |
| Key naming matches domain prefix convention | `registry.validate()` on every build + CI `build --validate-only` | CI build blocked |
| All variables declared and used (none extra, none missing) | `registry.validate()` тАФ non-empty errors = exit 1 | CI build blocked |
| Severity must be INFO/WARN/ERROR/CRITICAL | `registry.validate()` | CI build blocked |
| Surface must be subset of UI/AUDIT/NOTIFICATION | `registry.validate()` | CI build blocked |
| `hi:` stub field required on every key | `registry.validate()` | CI build blocked |
| `CTS_Msg_Taxonomy.html` committed alongside `messages.yaml` | pre-commit Check 11 | Commit blocked |
| `CTS_Msg_Taxonomy.html` never edited directly | pre-commit Check 12: staged without `messages.yaml` = WARNING | Commit warned |
