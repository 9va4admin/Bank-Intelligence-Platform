1. Architecture Overview
Design principles governing every Kafka topic and Redis key in ASTRA

Blast Isolation First

CTS and EJ never share a Kafka consumer group, Redis cluster, or database connection pool. A CTS surge cannot starve EJ. An EJ Redis failure cannot affect CTS. Hard boundaries, not conventions.

Exactly-Once Semantics

Every Kafka producer uses enable.idempotence=true + acks=all. NGCH filing is deduplicated via Temporal workflow IDs and Redis distributed locks. No cheque is ever filed twice.

IET Safety Above All

IET breach rate target: 0.000%. Every Redis operation in the CTS path is fail-open. Every Kafka consumer lag is monitored by KEDA and Grafana. An IETWatchdogWorkflow fires at T−30s regardless of any other system state.

Per-Bank Topic Partitioning

All high-throughput topics use {bank_id} as the Kafka message key, ensuring ordered processing per bank while enabling parallel processing across banks within a topic.

Cache-Aside + Write-Through

Redis vaults use cache-aside for reads and write-through for CBS-synced data. Config cache uses event-driven invalidation (Kafka → Redis DEL). AI outputs are never cached — every cheque is unique.

Kafka as Audit Trigger

Every terminal workflow state publishes to platform.audit.events. Immudb writes happen via a Redis Streams buffer — absorbing spikes and ensuring HSM signing happens once before append.

6. Redis Key Namespaces — Complete Reference
Every Redis key pattern used in ASTRA — with type, TTL, eviction policy, and purpose
sig:{bank_id}:{account_hash}
Signature Vault
redis-cts · Hash type · No TTL
sig:{bank_id}:{hmac_sha256(account_num)}
HASHNo TTL (evict on LRU)
Fields: vector_b64 (Siamese embedding, base64), specimen_count, last_synced_at, account_suffix (last 4 digits for display only). Key uses HMAC-SHA256 with bank-specific pepper from Vault — never raw account number.
sig:miss:{bank_id}:{account_hash}
STRINGTTL: 300s
Vault miss marker — set when CBS lookup also fails. Prevents repeated CBS calls for accounts with no specimen. Cleared by platform.cache.invalidation event after vault sync.
pps:{bank_id}:{account_hash}:{series}
Positive Pay Vault
redis-cts · Hash type · TTL = cheque validity
pps:{bank_id}:{account_hash}:{series_start}
HASHTTL: 90d (cheque validity)
Fields: payee_name_enc (pgcrypto encrypted), amount_range, valid_from, valid_until, status (ACTIVE/STALE). Series key enables O(1) lookup per cheque series.
pps:miss:{bank_id}:{account_hash}
STRINGTTL: 300s
PPS vault miss marker — account has no positive pay instruction. Routes to HUMAN_REVIEW. Never routes to AUTO_RETURN. Cleared on vault sync.
ratelimit:{bank_id}:{endpoint}:{minute}
API Rate Limiting
redis-cts · String type · TTL = 70s
ratelimit:{bank_id}:{endpoint_slug}:{epoch_minute}
STRING (counter)TTL: 70s
Sliding window counter per bank per endpoint per 60-second window. Atomic INCR + EXPIRE via Redis pipeline. Endpoint slugs: cts_inward_submit (600/min), cts_decisions_get (300/min), ej_inward_log (100/min), admin_config (30/min). Fail-open when Redis unavailable.
session:{bank_id}:{token_hash}
JWT Session Cache
redis-cts · String type · TTL = 900s (sliding)
session:{bank_id}:{sha256(jwt_token)}
STRING (JSON)TTL: 900s (refreshed on access)
Cached JWT claims — bank_id, user_id, roles, expires_at. Avoids repeated crypto verify (expensive SAML assertion check). Token stored as SHA-256 hash only — raw JWT never in Redis. Sliding expiry: EXPIRE reset on each request. DEL on logout.
lock:workflow:{bank_id}:{workflow_id}
Distributed Workflow Locks
redis-cts · String type · TTL = 60s
lock:workflow:{bank_id}:{instrument_id}
STRING (token)TTL: 60s (auto-release on crash)
Idempotency guard — prevents duplicate Temporal workflow submissions for the same instrument. SET NX PX atomic acquire. Lua script atomic check-and-delete release. Unique token per lock instance prevents cross-caller release. Fail-open when Redis None (Temporal handles dedup).
audit:{bank_id}:stream
Audit Event Stream Buffer
redis-cts · Stream type · MAXLEN 1000
audit:{bank_id}:stream
STREAM (XADD/XREADGROUP)MAXLEN 1000 (trimmed)
Redis Streams buffer between services and Immudb writer. Fields: event_type, entity_id, actor_id, bank_id, payload (JSON), hsm_signed ("true"/"false"). Consumer group: cg-audit-immudb-{bank_id}. XACK after Immudb append. HSM signing happens inside consumer before write. Absorbs burst writes — Immudb receives steady stream.
config:{bank_id}:{key}
Config Service L3 Cache
redis-cts · String type · TTL = 30s
config:{bank_id}:{config_key}
STRINGTTL: 30s (event-invalidated)
Layer 3 config values from YugabyteDB — thresholds, business rules. 30s TTL is the fallback. Primary invalidation: Kafka platform.config.changed → CacheInvalidator → Redis DEL. Workers reload within 30s of Admin UI change. Keys follow dot notation: iet_minutes, stp_auto_confirm_threshold, human_review_fraud_threshold.
config:idempotency:{bank_id}:{request_id}
STRINGTTL: 300s
API-level idempotency keys for cheque submission endpoints. Prevents double-submit from retry storms. Value: workflow_id of created workflow.
dashboard:cts:{bank_id}:*
CTS Dashboard Aggregates
redis-cts · Hash / Sorted Set · TTL = 60s
dashboard:cts:{bank_id}:session:{date}
HASHTTL: 60s
Real-time session aggregates: total_inward, stp_confirmed, stp_returned, human_review, iet_near_breach, avg_decision_ms. Updated by analytics-service consumer. Dashboard polls this — never hits YugabyteDB for real-time numbers.

Top fraud-risk accounts by SHAP score sum (rolling 24h). Sorted set enables O(log N) top-K queries. No PII in keys — account_hash only.

7. Redis Patterns — Implementation Reference
Reusable patterns used across ASTRA — with rationale and failure behaviour
Pattern: Vault Lookup (Cache-Aside)
1
Redis GET → Cache Hit
Hash lookup in redis-cts. Returns in <5ms. No CBS call.
HGET sig:{bank_id}:{account_hash} vector_b64
↓
2
Cache Miss → CBS Lookup
Call CBS connector via Temporal activity (CBS_TIMEOUT=10s)
check_cbs_balance() activity with CBS_RETRY policy
↓
3
CBS Hit → Write-Through Cache
HMSET vault entry, no TTL (daily sync updates it)
HMSET sig:{bank_id}:{hash} vector_b64 ... last_synced ...
↓
4
CBS Miss → Set Miss Marker + HUMAN_REVIEW
SET miss marker with 300s TTL. Route to human review. NEVER auto-return on vault miss.
SET sig:miss:{bank_id}:{hash} "" EX 300
Pattern: Rate Limiting (Sliding Window)
1
Extract bank_id from JWT + resolve endpoint slug
SHA-256 token → session cache lookup → bank_id claim. Compute minute window = int(time()) // 60.
key = ratelimit:{bank_id}:{slug}:{window}
↓
2
Atomic INCR + EXPIRE via Pipeline
Single round-trip: INCR returns new count, EXPIRE sets 70s window. No race condition.
PIPELINE: INCR key → EXPIRE key 70 → EXECUTE
↓
3a
count ≤ limit → Pass with headers
Set X-RateLimit-Limit, X-RateLimit-Remaining, X-RateLimit-Reset on response.
↓
3b
count > limit → 429 + Retry-After
JSON: {error_code: RATE_LIMIT_EXCEEDED, retry_after_seconds: ...}
↓
4
Redis error → Fail-Open
Rate limiting is DoS protection, not a safety gate. Request passes through.
Pattern: Distributed Lock (Workflow Idempotency)
1
Atomic Acquire
SET lock:workflow:{bank_id}:{wf_id} {token} NX PX 60000
Returns OK (acquired) or nil (already held). Token = uuid4 unique per instance.
↓
2
Start Temporal workflow in context
Lock held during workflow submission only — not for duration of workflow. Temporal handles long-running idempotency via workflow_id.
↓
3
Atomic Release (Lua)
if GET key == token then DEL key end
Lua script: check-and-delete is atomic. Token check prevents cross-caller release (expired lock re-acquired by another caller).
Pattern: Audit Stream Buffer (Redis Streams)
1
Service calls buffer_audit_event()
XADD audit:{bank_id}:stream MAXLEN 1000 * event_type ... payload ...
↓
2
Audit-service consumer reads batch
XREADGROUP GROUP cg-audit-immudb-{bank_id} consumer-1 COUNT 50 STREAMS audit:{bank_id}:stream >
↓
3
HSM sign → Immudb append
FIPS 140-2 HSM signs each event. Immudb Merkle tree append. Cryptographically tamper-evident.
↓
4
XACK removes from PEL
XACK audit:{bank_id}:stream cg-audit-immudb-{bank_id} {msg_id}
Only after Immudb confirms write. Unacked messages redelivered on consumer crash.

How Kafka and Redis collaborate in the most latency-sensitive paths
Flow 1: Inward Cheque → Decision (<600ms wall clock)
1
NGCH → Kafka (~2ms)
PRODUCE cts.inward.{bank_id} key={bank_id}
2
KEDA detects lag → scale cts-agent-worker (~500ms if cold)
Warm pool (min=2 pods) means lag=0 during clearing window. Scale-up only for burst.
3
Temporal workflow starts — IETWatchdog spawned first (~5ms)
4
Vault lookup (Redis GET) (<5ms)
HGET sig:{bank_id}:{hash} vector_b64
5
OCR + Signature + Fraud Score in parallel (~300ms — GPU)
6
File to NGCH → Publish cts.decisions + platform.audit.events
PRODUCE platform.audit.events → Redis Streams buffer
✓
Total wall clock: <600ms (p99 target)
Redis vaults: <5ms. GPU inference: ~300ms. Kafka produce: ~5ms. NGCH file: ~100ms.
Flow 2: Config Change → All Workers Reload (<30s)
1
Admin UI maker-checker → config_service.publish()
Layer 3 change approved. config_service writes to YugabyteDB first.
2
Kafka publish: platform.config.changed
PRODUCE platform.config.changed {event_type: CONFIG_THRESHOLD_CHANGED, key: "iet_minutes", bank_id: ...}
3
CacheInvalidator (api-gateway) consumes → Redis DEL
DEL config:{bank_id}:iet_minutes
4
cts-agent-worker consumes → reload from YugabyteDB
Next config_service.get() call fetches fresh value from DB, caches in Redis with new 30s TTL.
✓
All workers updated within 30 seconds
No pod restart. No deployment. No ASTRA involvement. Bank ops team makes change, workers react.

1
NPCI claim arrives → ej.disputes.{bank_id}

Dispute engine: Redis cache check for dedup
GET ej:dispute:status:{bank_id}:{claim_id}

Claim text embedded. Cosine similarity vs EJ canonical records. Top-3 matches returned.

Cache result → Redis. Publish resolution
SET ej:dispute:status:{bank_id}:{claim_id} AUTO_RESOLVED EX 259200

10. Blast Isolation — CTS ↔ EJ Hard Boundaries
What is physically separated and why — zero shared resources between modules
Layer CTS Resource EJ Resource Isolation Guarantee
Kubernetes namespace: astra-cts-{bank_id} namespace: astra-ej-{bank_id} ResourceQuota + LimitRange per namespace. CTS cannot consume EJ CPU budget.
Redis redis-cts.{bank_id}.internal:6379 redis-ej.{bank_id}.internal:6379 Separate Helm releases, separate node pools. CTS eviction pressure cannot evict EJ keys. Separate memory budgets.
Kafka Topics cts.* (9 topics) ej.* (7 topics) Separate consumer groups. CTS workers never subscribe to ej.* topics. KEDA ScaledObjects are independent.
Temporal task_queue: cts-processing-{bank_id} task_queue: ej-normalisation-{bank_id} Separate worker Deployments. A saturated CTS task queue does not delay EJ normalisation.
Database pgbouncer-cts (schema: cts) pgbouncer-ej (schema: ej) Separate connection pools. A CTS connection storm cannot exhaust EJ connections. Schema-level row isolation.
vLLM / GPU queue: cts-vision, cts-ocr queue: ej-reasoning, ej-embeddings Separate vLLM worker processes per queue. CTS vision GPU saturation does not delay EJ LLM parsing.
Python Imports modules/cts/** modules/ej/** Cross-import is FORBIDDEN. Semgrep CI rule blocks merge. Only shared/ is common ground.
Shared Services (Allowed Exceptions)
Three services are intentionally shared but have per-module rate limits and separate logical partitions:
audit-service
Shared, but CTS writes to Immudb collection cts_events and EJ writes to ej_events. Separate Redis Stream buffers per module.
notification-service
Shared, but separate Kafka consumer groups: cg-notify-cts and cg-notify-ej. Email + WhatsApp channels are stateless.
analytics-service
Read-only consumer with separate consumer groups per module. No write path — purely async. Cannot cause write contention between modules.
