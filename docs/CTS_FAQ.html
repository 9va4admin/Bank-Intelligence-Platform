Platform Overview & Architecture
▾ What is ASTRA CTS and what problem does it solve? CTORBI

ASTRA CTS (Cheque Truncation System) is a banking-grade AI platform that automates India's RBI-mandated cheque clearing process end-to-end — both the inward clearing (drawee bank role) and outward clearing (presentee bank role).

The primary driver is the RBI IET (Item Expiry Time) mandate effective January 2026: a drawee bank has exactly 180 minutes (T+3 hours) from the time a cheque arrives at NGCH to file a decision. A missed IET is a deemed approval — the bank must pay regardless of fraud. With 500+ cheques arriving simultaneously, human processing alone cannot guarantee the deadline. ASTRA deploys one AI agent per inward cheque, running 500 agents in parallel, and delivers each decision in under 600ms wall clock.

For outward clearing, ASTRA automates scanner capture → MICR extraction → CTS-2010 image compliance → lot creation → endorsement stamping → NGCH submission → session reconciliation, replacing fragmented manual workflows.

ASTRA is an on-premises platform. No cheque image, no customer data, and no decision ever leaves the bank's own data centre.

▾ Is ASTRA a SaaS platform or on-premises? CTOInfoSec

Strictly on-premises. ASTRA is deployed inside the bank's own data centre, within the bank's own Kubernetes cluster, using the bank's own servers, network, and storage. There is zero cloud dependency.

  • No customer data travels outside the bank's perimeter — ever
  • No ASTRA vendor has shell or kubectl access to any bank's production cluster
  • All software updates are pulled by the bank's own ArgoCD from a versioned OCI Helm registry — the bank controls when upgrades apply
  • AI models (Qwen2-VL, Llama 3.3, GOT-OCR2) run on the bank's own GPU servers — no inference call reaches the internet
This architecture satisfies RBI IT Framework data localisation requirements and SEBI cloud policy constraints for core banking operations.

▾ Can ASTRA run in a multi-data-centre active-active configuration? CTORBI

Yes. ASTRA is designed for active-active across two data centres — both DCs serve live traffic simultaneously. A third air-gapped DC is used for backup only (not in the traffic path).

  • RPO = 0 — zero data loss on DC failure (YugabyteDB active-active replication + Kafka MirrorMaker 2 bidirectional sync)
  • RTO < 30 seconds — automatic failover; no manual intervention
  • If DC1 fails during a clearing session, DC2 picks up all Kafka consumer lag and continues processing in under 30 seconds
  • IETWatchdogWorkflow survives DC failure — the watchdog is durable in Temporal and resumes from checkpoint on the surviving DC
Single-DC deployments are also supported for pilot banks. Active-active is recommended before go-live for production clearing volumes.

Inward Clearing (Drawee Bank Role)
▾ How does ASTRA process an inward cheque from NGCH arrival to decision? OpsRBI

The full pipeline runs inside a single ChequeProcessingWorkflow (Temporal), executing these activities in sequence:

  1. IET Watchdog spawned first — a child workflow starts immediately, counting down from T to T-30 seconds; if no decision is filed by T-30s, it files an automatic decision to NGCH before IET breach
  2. CTS-2010 Compliance Validation — image resolution, orientation, MICR band checks
  3. OCR Extraction — GOT-OCR2.0 extracts MICR line (IFSC, account, cheque number, amount)
  4. Alteration Detection — Qwen2-VL 72B analyses image for overwriting, erasure, ink inconsistency
  5. Signature Verification — Siamese network compares live signature against vault specimen
  6. PPS Lookup — Positive Pay System check from Redis vault
  7. CBS Balance + Account Status — live check: account open/frozen/closed, available balance, stop-payment instructions
  8. Fraud Scoring — XGBoost ensemble + SHAP explainer
  9. Decision Synthesis — LLM (Llama 3.3 70B) reviews all signals and produces STP_CONFIRM / STP_RETURN / HUMAN_REVIEW with full rationale
  10. File to NGCH — decision submitted via ngch_filer; idempotency key ensures no duplicate filing
  11. Audit Write — every activity's result written to Immudb (HSM-signed)

Total wall-clock time for 500 simultaneous cheques: <600ms p99. Each cheque is an independent workflow; they run fully in parallel.

▾ What happens when a cheque is flagged for human review? Ops

When the AI decision is HUMAN_REVIEW, a HumanReviewWorkflow is triggered:

  • The cheque appears in the Ops Workstation review queue with full context: cheque image, SHAP values, fraud score, OCR output, signature match score, CBS account status, alteration findings
  • The ops_reviewer (authorised by zone) has a maximum 55-minute window to decide (Confirm or Return)
  • If no reviewer acts within 55 minutes, the workflow auto-returns the cheque — this is a safe conservative default that prevents IET breach while the IETWatchdogWorkflow continues to monitor
  • The reviewer's decision is filed to NGCH immediately via the same ngch_filer activity
  • All reviewer actions are logged to Immudb with reviewer identity, timestamp, and decision rationale
The 55-minute human review window is calibrated against the IET window. Banks with tighter SLAs can reduce this via Admin UI (Layer 3 config) without a code change.
▾ Can ASTRA process 500 cheques simultaneously? What is the infrastructure requirement? CTOOps

Yes. ASTRA's design principle is one agent per cheque. When 500 cheques arrive in a Kafka batch, 500 independent Temporal workflows spawn via KEDA auto-scaling.

Scale-up speed: KEDA triggers on Kafka consumer lag >10 messages. It can scale from 2 to 500 pods in under 2 seconds (aggressive +50 pods/30s policy).

GPU minimum for pilot (quantised models):

  • 4× NVIDIA RTX 4090 — runs Qwen2-VL 7B + Llama 3.1 8B quantised (AWQ)

GPU target for production (full-size models):

  • 4–8× NVIDIA A100 80GB — runs Qwen2-VL 72B + Llama 3.3 70B quantised (GPTQ)

CPU-bound activities (PPS vault, CBS check, fraud scoring) scale on standard Kubernetes nodes. GPU is only required for vision and reasoning LLMs.

▾ What happens if the CBS is unreachable when a cheque arrives? OpsCTO

ASTRA follows a strict graceful degradation policy. If CBS is unreachable:

  • The check_cbs_balance activity retries 3 times with exponential backoff (2s, 3s, 4.5s)
  • If still unreachable, the activity returns CbsResult(available=None, status="UNREACHABLE") — it does not crash the workflow
  • The decision engine proceeds with image-only processing (OCR + alteration + signature + PPS)
  • Without balance confirmation, the cheque is routed to HUMAN_REVIEW (not auto-confirmed or auto-returned)
  • The IETWatchdogWorkflow continues counting down — the cheque will still be filed before IET breach
ASTRA never silently returns an incorrect value (e.g. balance=0) on CBS timeout. The status is explicitly UNREACHABLE and the degraded path is activated.

Outward Clearing (Presentee Bank Role)
▾ How does outward clearing work in ASTRA? Ops

Outward clearing covers cheques your customers deposit (drawn on other banks). The workflow:

  1. Scanner Capture (OutwardScanWorkflow) — branch scanner connects to ASTRA; images captured and MICR line extracted in real time
  2. CTS-2010 Image Compliance — image validated for NGCH acceptance (resolution, orientation, MICR band); rejected images flagged for re-scan
  3. Lot Creation — accepted instruments grouped into lots by drawee bank and clearing zone
  4. Lot Sealing — ops manager seals the lot; triggers BatchEndorsementWorkflow
  5. Endorsement Stamping (BatchEndorsementWorkflow) — digital endorsement applied to each instrument in the lot per RBI CTS 2010 spec
  6. NGCH Submission (NGCHSubmissionWorkflow) — endorsed lot submitted to NGCH when clearing session opens; acknowledgement receipt confirmed and logged
  7. Session Reconciliation (SessionReconciliationWorkflow) — at session close, NGCH settlement report matched against submitted instruments; unmatched items flagged; Return Reason File (RRF) generated
▾ What is the Presentment File / CXF file and how is it generated? OpsNPCI

The Presentment File (CXF — CTS eXchange Format) is the structured file submitted to NGCH containing all outward instruments for a clearing session. ASTRA generates it automatically at the end of the BatchEndorsementWorkflow.

Contents: header record (presenting bank IFSC, date, session ID, item count), one detail record per instrument (drawee IFSC, account, cheque number, amount, image hash, MICR line), and a trailer record with control totals.

The file is digitally signed using the bank's HSM-held PKI certificate before submission. A separate Rejected Instruments File captures CTS-2010 non-compliant items with rejection reason codes.

Ops staff can preview both files in the ASTRA workstation UI before submission, and download them for records.

▾ How does ASTRA handle return reason files (RRF) from NGCH? Ops

After each clearing session, NGCH sends a Return Reason File (RRF) listing instruments that were dishonoured by drawee banks. ASTRA's SessionReconciliationWorkflow:

  • Downloads the RRF from NGCH (or processes webhook notification in Phase 2 of NGCH modernisation)
  • Matches each returned instrument against the submitted lot
  • Updates instrument status in YugabyteDB
  • Generates a customer-facing return notice (routed through Notification Service — email or WhatsApp depending on bank configuration)
  • Flags unmatched items for ops_manager review
  • Writes full reconciliation outcome to Immudb

Reconciliation results are visible in the Ops Workstation under the "Session Reconciliation" view, with drill-down to individual returned instruments.

▾ Which physical scanner brands does ASTRA support? OpsCTO

ASTRA uses a pluggable scanner adapter architecture. The modules/cts/scanner/adapters.py layer abstracts scanner-specific SDKs. Out-of-the-box adapters are provided for all CTS-compliant scanner models certified by NPCI/RBI for CTS-2010 scanning (Canon, Panini, Digital Check, Burroughs-class devices).

Adding a new scanner requires implementing the ScannerAdapter base class — typically a 1–2 day integration effort by ASTRA's professional services team. The adapter handles image capture, MICR line reading (E-13B font), and quality pre-check before image reaches ASTRA's compliance engine.

IET Mandate & RBI Compliance
▾ What is the IET mandate and what are the consequences of a breach? RBICTO

The Item Expiry Time (IET) is an RBI mandate (effective January 2026) requiring a drawee bank to file a decision on every inward cheque within 180 minutes (T+3 hours) of its arrival at NGCH.

Consequences of IET breach:

  • The cheque is treated as a deemed approval — NGCH processes it as honoured
  • The bank must pay the cheque amount regardless of fraud, insufficient funds, or stop payment
  • RBI can levy penalties on the bank for systematic IET breach patterns
  • Reputational risk with NPCI / RBI if breach rates are above zero
ASTRA's IET breach target is 0.000% — not 99.9%, not 99.99%. Zero. This is enforced architecturally, not by SLA language.
▾ How does ASTRA guarantee zero IET breaches? RBICTO

IET safety is structural — built into the architecture, not a policy statement:

  1. IETWatchdogWorkflow spawned first — before any processing activity starts, a child Temporal workflow is spawned with the instrument's IET deadline. Even if the parent workflow crashes, the watchdog continues independently (ParentClosePolicy = ABANDON)
  2. T-30 second emergency filer — at T-30 seconds, if no decision has been filed, the watchdog automatically files a return decision to NGCH. This ensures a decision is always on record before IET
  3. Temporal durability — workflow state persists across pod restarts, DC failures, and network partitions; the countdown resumes from exactly where it left off
  4. Active-active DC — if one DC goes down, the watchdog workflow resumes on the surviving DC within 30 seconds
  5. NGCH SFTP fallback — if the primary NGCH transport is unavailable, the watchdog uses the backup SFTP path; both are always available until NPCI decommissions SFTP (30-month notice required)
  6. Exactly-once filing — idempotency keys prevent duplicate NGCH submissions even if the watchdog and the main workflow both try to file

The 180-minute IET window is configurable per bank via Admin UI (iet_minutes). The watchdog T-30s trigger is a fixed architectural constant — it cannot be turned off by any bank configuration.

▾ Can RBI examiners view IET compliance records? RBI

Yes. ASTRA includes an rbi_examiner role with read-only, time-scoped access:

  • Access is provisioned by the bank IT admin for a specific audit engagement period (start date, end date)
  • The examiner can view all IET timing records: arrival timestamp, decision timestamp, filing timestamp, IET window utilised
  • Every cheque's complete processing timeline is available in the Audit Console
  • Immutable audit records in Immudb include cryptographic proof (Merkle tree verification) that records have not been tampered with since creation
  • The examiner cannot modify any data, cannot access customer PII beyond masked displays, and cannot extend their own access period

Access sessions are logged to Immudb — the bank has a complete record of what the examiner viewed and when.

▾ Does ASTRA comply with RBI IT Framework controls? RBIInfoSec

ASTRA is designed against the RBI IT Framework for Banks (2011 + 2021 amendments) and the RBI Master Direction on IT (2023). A detailed control mapping document (compliance/rbi-it-framework/control-mapping.yaml) maps every ASTRA architectural element to the relevant RBI control.

Key control areas covered:

  • Data Localisation — all data stays within the bank's network; no cloud processing
  • Audit Trail — Immudb provides tamper-evident, cryptographically verifiable audit logs
  • Access Control — SAML 2.0 + RBAC + ABAC; no passwords stored by ASTRA; principle of least privilege
  • Encryption — AES-256 at rest (MinIO SSE-KMS), TLS 1.3 in transit, column-level pgcrypto for PII
  • HSM for PKI — FIPS 140-2 Level 3 for all NGCH signing keys
  • Business Continuity — active-active DC, RPO=0, RTO<30s
  • Change Management — all config changes via maker-checker; Immudb-audited before taking effect

AI Decision Engine
▾ Is ASTRA's AI a black box? How do we explain a decision to a customer or regulator? RBICTOInfoSec

ASTRA uses fully explainable AI. Every decision has three layers of explainability:

  1. SHAP values — each feature's contribution to the fraud score is quantified (e.g. "account_age_days contributed −0.12 to fraud probability; recent_stop_payment contributed +0.31")
  2. LLM narrative rationale — Llama 3.3 70B synthesises all signals into a human-readable decision summary (e.g. "Signature match score 0.91 exceeds threshold. No PPS record on file — routed to human review per bank policy.")
  3. Activity-level audit trail — every activity's input and output is logged to Immudb; an examiner can reconstruct exactly what each AI model saw and what it concluded
No AI decision is ever filed to NGCH without SHAP values computed and stored. This is a platform invariant enforced in code, not a policy.

For customer dispute responses, ops_managers can generate a structured decision summary report from the Audit Console that is safe to share (PII masked, decision trail included).

▾ What happens when the AI models are unavailable (GPU server down)? OpsCTO

ASTRA's graceful degradation hierarchy for AI unavailability:

  1. Retry — each AI activity retries up to 2× with exponential backoff
  2. Rule-based fallback — if vLLM remains unavailable, the decision engine switches to a deterministic rule-based scorer (no GPU required)
  3. Human review escalation — lower confidence decisions route to the human review queue
  4. IET watchdog continues — even with all AI down, the watchdog ensures every cheque is filed before IET breach; the decision filed may be a conservative return or human-review outcome
ASTRA never silently fails. It never files a random decision. And it never breaches IET because the AI is unavailable.
▾ How is the signature verification model trained? On whose data? InfoSecCTO

The Siamese neural network for signature verification is trained exclusively on the bank's own specimen signatures from their CBS/signature card records. No cross-bank training data is used — each bank's model is a separate instance trained on that bank's customer population only.

Training process:

  • Initial training on the bank's existing signature card scans (typically millions of specimens)
  • Fine-tuning on the bank's historical CTS cheque dataset (where available) with verified genuine/forged labels
  • Model is versioned in MLflow; every deployment is tracked with training data hash, training date, and accuracy metrics
  • No customer signature data leaves the bank during training or inference

Signature specimens are stored in the Redis Signature Vault, hashed with HMAC-SHA256 using a bank-specific Vault-managed pepper — raw account numbers never appear as vault keys.

SB vs SMB Routing (Sponsor Bank & Sub-Member Banks)
▾ What is the SB / SMB distinction in CTS clearing? OpsNPCI

In CTS clearing, not all banks are direct members of NGCH. Smaller banks (typically Urban Co-operative Banks, RRBs) participate through a Sponsor Bank (SB):

TypeFull NameNGCH RelationshipExample
SBSponsor BankDirect NGCH member; submits for itself AND all its SMBsSaraswat Co-operative Bank
SMBSub-Member BankRoutes instruments through SB; not a direct NGCH memberA smaller UCB in Maharashtra

ASTRA handles both roles natively. A bank deploying ASTRA as an SB gets the full SMBForwardingWorkflow that routes SMB instruments into the SB's lots and submits them to NGCH on behalf of the SMBs. SMBs deploying ASTRA get a scoped view of only their own instruments.

▾ How does an SMB instrument flow through an SB's ASTRA deployment? OpsNPCI

When an SMB's instrument arrives at the SB (via the cts.smb.inbound.{bank_id} Kafka topic), the SMBForwardingWorkflow executes:

  1. Validate SMB instrument — CTS-2010 compliance; SMB IFSC verification; risk shield check (stops fraudulent bulk submissions from an SMB)
  2. Route to sponsor lot — the instrument is added to the SB's outward lot for the appropriate clearing zone and drawee bank
  3. Forward to NGCH — submitted as part of the SB's NGCHSubmissionWorkflow; NGCH sees it as the SB's submission
  4. Notify SMB — submission acknowledgement sent back to the SMB bank's ops team
  5. Write audit — full event trail in Immudb with SMB identity, instrument reference, lot assignment

The SB Ops team can view all SMB instruments in a consolidated view, or filter by SMB in the Presentment File screen. Download files can be generated per-SMB for settlement reconciliation.

▾ How does ASTRA's Risk Shield protect the SB from fraudulent SMB submissions? InfoSecOps

The SMB Risk Shield in modules/cts/sub_member/risk_shield.py protects the SB from liability for fraudulent or erroneous SMB instrument batches:

  • Volume anomaly detection — sudden spike in SMB submission volume triggers hold and ops_manager review
  • IFSC verification — SMB instruments must carry the SMB's own registered IFSC; mismatches are rejected immediately
  • Duplicate detection — instrument IDs checked against a rolling Redis window; duplicates from an SMB are rejected and logged
  • Risk scoring per SMB — each SMB has a configurable risk profile; high-risk SMBs require ops_manager sign-off before lot inclusion
  • Automatic quarantine — if an SMB exceeds anomaly thresholds, all subsequent submissions from that SMB are quarantined pending SB ops_manager review

Vault & Signatures
▾ What is the Signature Vault and how is it kept up to date? InfoSecOps

The Signature Vault is a Redis cluster (redis-cts) that caches the vector embeddings of every customer's registered specimen signature. When a cheque arrives, the Siamese network compares the live signature against the vault embedding — the lookup takes <5ms.

How it stays current:

  • VaultSyncWorkflow runs daily at 6AM — pulls all new/updated signature specimens from the CBS and refreshes the Redis vault
  • Real-time events — the CBS Connector publishes signature change events to the cts.vault.sync.{bank_id} Kafka topic; the vault is updated within seconds of a CBS change (new account, new specimen)
  • Vault entries have no TTL — they persist until explicitly refreshed by the sync workflow

Vault keys never contain raw account numbers. Keys are formatted as sig:{bank_id}:{HMAC-SHA256(account_number)} using a Vault-managed pepper — the mapping from HMAC to account is not stored in Redis.

▾ What happens when a signature is not found in the vault (vault miss)? OpsRBI

A vault miss (signature or PPS key not found in Redis) always routes the cheque to HUMAN_REVIEW. This is a non-negotiable platform invariant:

ASTRA never auto-returns a cheque solely because the vault has no record. A vault miss could mean a new account, a vault sync delay, or a data gap — not necessarily fraud. Human review is the only safe outcome.

This is enforced at three levels:

  • Code: the vault miss handler always returns HUMAN_REVIEW — this path is tested at 95%+ coverage
  • Config: vault_miss_action is locked to HUMAN_REVIEW in Layer 1 (Helm chart) — no bank Admin UI can change this to AUTO_RETURN
  • Audit: every vault miss is logged as a WARN event in Immudb with instrument ID and timestamp
▾ What is the Positive Pay System (PPS) vault? Ops

The PPS Vault is a Redis cluster entry set per registered Positive Pay instruction. When a customer issues a high-value cheque and registers it via Positive Pay (amount, payee, date, cheque number), the CBS writes a PPS entry. ASTRA checks the arriving inward cheque against this entry:

  • PPS match — amount, payee, and cheque number match the registered instruction → one signal in favour of STP_CONFIRM
  • PPS mismatch — amount or payee differs from the registered instruction → strong signal for STP_RETURN (reason: "PPS mismatch")
  • PPS not registered (vault miss) → routes to HUMAN_REVIEW if the cheque is above the configured high_value_amount_threshold

PPS vault key format: pps:{bank_id}:{HMAC(account_number)}:{cheque_series_start}. TTL = cheque validity period.

▾ How are secrets (passwords, API keys, certificates) managed? InfoSec

Zero secrets in code or git. Every credential is managed by HashiCorp Vault (on-premises) with these properties:

  • Dynamic secrets — database passwords, Redis tokens, and CBS credentials are rotated automatically every 24 hours; old credentials expire and cannot be reused
  • No static credentials — ASTRA applications never have a "password" that persists; they receive a short-lived dynamic credential from Vault on startup
  • Vault agent sidecar — each pod receives its Vault token at startup via a Kubernetes sidecar injection; the application code never authenticates to Vault directly
  • HSM for PKI — NGCH signing keys are held in a FIPS 140-2 Level 3 Hardware Security Module; private keys never exist in software
  • gitleaks enforcement — pre-commit hooks and CI scan every file for credential patterns; any hardcoded secret blocks the commit and the PR
▾ What network security controls does ASTRA enforce between services? InfoSecCTO

ASTRA uses Istio service mesh for zero-trust networking:

  • mTLS everywhere — every pod-to-pod communication uses mutual TLS; Istio issues certificates automatically via cert-manager; no service can impersonate another
  • Istio AuthorizationPolicy — CTS pods cannot call EJ pods directly; EJ pods cannot call CTS pods; only explicitly permitted paths are allowed
  • Namespace isolation — CTS and EJ run in separate Kubernetes namespaces with ResourceQuotas; a resource exhaustion in one cannot affect the other
  • No direct DB access — applications connect through pgbouncer connection pools; direct YugabyteDB access from application pods is blocked by NetworkPolicy
  • Istio Ingress Gateway — all external traffic enters through a single controlled ingress; internal services are never exposed externally
  • TLS 1.3 minimum — older TLS versions are rejected by Istio configuration; cannot be downgraded by any bank setting
▾ Does ASTRA support penetration testing? What is the pen test scope? InfoSec

Yes. ASTRA is designed for bank-commissioned penetration testing and supports it in a dedicated staging environment (not production). Recommended pen test scope:

  • API gateway authentication — JWT forgery, session token manipulation
  • RBAC bypass attempts — accessing data outside role or zone scope
  • NGCH webhook endpoint — HMAC verification bypass, replay attacks
  • SQL/NoSQL injection in API endpoints
  • mTLS bypass attempts at Istio ingress
  • Vault path traversal — attempting to read secrets outside authorised paths
  • Immudb write bypass — attempting to write audit events without HSM signing
  • AI decision manipulation — adversarial image inputs to vision models

A security test harness (OWASP ZAP configuration) is included in the ASTRA release package. Full DAST scan results against the staging environment are required before every production release.

▾ How does ASTRA's immutable audit trail work? RBIInfoSec

ASTRA uses Immudb — a cryptographically tamper-evident database — for all audit events:

  • Every AuditEvent is HSM-signed (FIPS 140-2 Level 3 key) before writing to Immudb
  • Immudb uses a Merkle tree structure — any attempt to modify a historical record invalidates the tree hash, making tampering immediately detectable
  • Audit events are also written to MinIO WORM buckets in parallel — compliance-mode object lock ensures they cannot be deleted or overwritten for 10 years
  • An RBI examiner or internal auditor can request a Merkle consistency proof for any date range, verifying that the audit trail has not been modified since the initial write

What is audited: every cheque decision (all activities), every vault operation, every configuration change, every user login and action, every NGCH filing, every AI inference result, every diagnostic access session.

▾ Does ASTRA store user passwords? How does authentication work? InfoSec

ASTRA never stores passwords. Authentication is delegated entirely to the bank's existing Identity Provider (IdP) via SAML 2.0:

  • User logs into the bank's SSO portal (AD FS, Okta, or similar)
  • The bank's IdP issues a SAML assertion containing the user's identity, bank_id, roles, and clearing_zone attributes
  • ASTRA's API gateway validates the SAML assertion and issues a short-lived JWT (15-minute TTL) for internal use
  • The JWT is cached in Redis CTS cluster (not in the bank's IdP) for the session

This means: the bank's existing password policy, MFA requirements, account lockout rules, and session management all apply automatically. ASTRA adds zero new authentication attack surface.

RBI examiner accounts are provisioned by the bank IT admin for a specific access period — the bank's IdP governs the actual credential; ASTRA only governs the access scope and audit trail.

▾ How is maker-checker enforced for configuration changes? OpsRBI

All Layer 3 (business rule) configuration changes require dual approval:

  1. Maker (ops_manager) submits the proposed change in the Admin UI (e.g. "Change STP threshold from 0.92 to 0.94")
  2. Change is written to a pending queue in YugabyteDB — not yet active
  3. Checker (bank_it_admin) reviews and approves or rejects in the Admin UI
  4. On approval, the AuditEvent is written to Immudb first (with both maker and checker identities)
  5. Only after the Immudb write succeeds does the config publish to the platform.config.changed Kafka topic
  6. Worker pods reload the new value within 30 seconds — no restart needed
The maker and checker must be different users. The system enforces this — an ops_manager cannot approve their own submitted change.

Layer 4 (OPA Rego policy) changes require compliance_officer to author and bank_it_admin to approve — same dual-approval model, full diff stored in YugabyteDB.

Settlement & Reconciliation
▾ How does ASTRA handle end-of-day session reconciliation? Ops

At the close of each clearing session, ASTRA's SessionReconciliationWorkflow automatically runs:

  1. Fetch NGCH settlement report — downloads the session's final settlement file from NGCH
  2. Match submitted vs settled — every instrument submitted in the session is matched against NGCH's settlement confirmation; unmatched items are flagged
  3. Generate Return Reason File (RRF) — instruments returned by drawee banks are compiled into the RRF with standardised reason codes
  4. Write audit — full reconciliation record in Immudb
  5. Flag exceptions — any submitted instrument not found in NGCH's settlement report raises an exception for ops_manager review

Reconciliation results are available in the Ops Workstation under the "Session Reconciliation" tab with per-lot, per-drawee-bank, and per-SMB breakdowns.

▾ How does ASTRA support multi-centre (multi-city) bank operations? CTOOps

Large banks with Regional Processing Centres (RPCs) across multiple clearing zones are natively supported:

  • Each RPC connects independently to its zone's NGCH grid — no cross-zone consolidation needed at submission time (NPCI handles cross-zone settlement)
  • A Central Intelligence Hub aggregates reporting across all RPCs for consolidated dashboards, settlement position, and cross-centre fraud signals
  • Signature vault lookups are hub-and-spoke: RPC checks local Redis → on miss, queries Central Hub → result cached locally for subsequent lookups
  • PPS vault is always hub-and-spoke — PPS instructions can be submitted from any channel (branch, CBS, mobile) and must be consistently available at all RPCs

▾ How does ASTRA connect to NGCH today, and will it support NPCI's upcoming REST API? NPCICTO

  • Phase B — Webhook push (NPCI pushes inward cheques instead of SFTP poll); ASTRA adds a webhook receiver service; inward latency drops from 15-minute poll to real-time
  • Phase C — MCP Intelligence Layer (NPCI exposes an MCP server; ASTRA becomes an MCP client); deepest integration with real-time IET risk signals

The SFTP path is never decommissioned until NPCI provides 30 months' notice — it remains as a fallback throughout all phases.

▾ How are software upgrades applied? Can we roll back? CTOInfoSec

Upgrade process (bank-controlled, zero vendor access):

  1. ASTRA releases a new Helm chart version to the private OCI registry with release notes
  2. Bank raises a Change Request (bank's own ITSM), reviews release notes, gets CAB approval
  3. Bank IT admin changes targetRevision in ArgoCD from old version to new version
  4. Helm pre-upgrade hook runs: Alembic database migration (with dry-run report)
  5. Helm upgrade applies: rolling update, zero downtime for stateless services
  6. Temporal workers drain existing workflows gracefully before restart
  7. Post-upgrade hook runs automated smoke tests (Vault, mTLS, Immudb write)

Rollback: ArgoCD reverts targetRevision to the previous version. Alembic downgrade migration runs automatically. SLA: rollback complete in <10 minutes. All migrations are backward-compatible for one version — rollback never requires a data migration.

ASTRA never pushes upgrades to a bank. The bank pulls upgrades on its own schedule. No ASTRA engineer has access to any bank's production cluster.

▾ What happens to in-flight cheque workflows if a data centre fails mid-clearing? CTORBI

Temporal's durable workflow execution is the key mechanism:

  • Workflow state is replicated to both DCs via YugabyteDB active-active replication (RPO=0 — zero data loss)
  • On DC1 failure, Temporal workers on DC2 automatically resume all interrupted workflows from their last durable checkpoint — no manual intervention
  • IETWatchdogWorkflow continues counting down on DC2 — the IET deadline was already persisted before the DC failed
  • Kafka MirrorMaker 2 replicates all unprocessed inward cheque events bidirectionally — DC2 processes any events that DC1 had not yet consumed
  • Total recovery time: <30 seconds

ASTRA validates this behaviour continuously using Chaos Mesh — DC failure scenarios are part of the standard chaos test suite that runs in the staging environment before every production release.

▾ What is the backup strategy? InfoSecCTO
  • YugabyteDB — active-active replication provides real-time backup to the second DC; PITR (point-in-time recovery) snapshots taken every hour; snapshots sent to the air-gapped DC3
  • Kafka — topic data replicated bidirectionally via MirrorMaker 2; DC3 stores compressed topic snapshots daily
  • MinIO (cheque images, EJ files) — 3-copy erasure coding within each DC; cross-DC replication to DC2; WORM-locked cold tier in DC3
  • Immudb (audit trail) — audit events are written to Immudb on both DCs simultaneously; snapshots to DC3 daily
  • Redis (vaults) — persistence via AOF (append-only file) to NVMe; vault state rebuilt from YugabyteDB on Redis failure; not backed up to DC3 (it's a cache of CBS data)

DC3 is air-gapped — it cannot be reached from either production DC, so a ransomware attack on DC1/DC2 cannot spread to backups.
