The CTS ecosystem has five distinct participant classes — each with different capabilities, latency sensitivities, and IET exposure profiles. A modernisation proposal must serve all five simultaneously, without forcing any participant to abandon existing infrastructure before they are ready.

🏦 Drawee Bank (Payer's Bank)
  • Role: Pays or rejects the cheque
  • IET exposure: Maximum — must decide within 3 hours
  • Current pain: Learns of inward items only at SFTP poll cycle
  • Vision 2030 gain: Instant webhook delivery, AI-assisted decision in <600ms
🏦 Presentee Bank (Payee's Bank)
  • Role: Collects cheque, submits to NPCI
  • IET exposure: Low — submits and waits
  • Current pain: No per-instrument confirmation; batch failures are opaque
  • Vision 2030 gain: Synchronous receipt, per-instrument error codes
🏛 NPCI / NGCH (Clearing House)
  • Role: Routes instruments, manages settlement
  • Current state: SFTP-based batch hub
  • Vision 2030 role: REST API gateway → Webhook dispatcher → MCP intelligence server → Mobile clearing orchestrator
🏛 RBI (Settlement Authority)
  • Role: Holds inter-bank settlement accounts; enforces IET mandate
  • IET mandate: Jan 2026 — silence within 3 hours = deemed confirmation
  • Vision 2030 role: Regulatory sandbox for Phase 4 lien-mark mechanism; legal instrument framework for mobile-submitted cheques under NI Act 1881
👤 Drawer (Cheque Writer) — New Actor in Phase 4
  • Role (today): Passive — writes cheque, has no digital interaction post-issuance
  • Role (Phase 4): Active initiator — uploads scan, fills beneficiary data, receives lien notification
  • Requirement: RBI regulatory sandbox + NI Act clarification for mobile-originated instruments

How Cheque Clearing Works Today — The 10-Step SFTP Batch Process
Status Quo · T+1 Day · Batch File Model · No Real-Time Feedback

The Structural Reality: Today's CTS is a store-and-forward file transfer system dressed as a clearing network. Every actor in the chain polls for updates, waits for batch cycles, and has zero real-time visibility. Under a 3-hour IET mandate, this is structurally incompatible with safe, compliant clearing.

1
Payee Deposits Cheque at Their Bank (Presentee Bank)
The person who received a cheque (payee/beneficiary) deposits it at their bank branch. The cheque is drawn on the drawer's account at a different bank (the drawee bank). Same-bank cheques — where drawer and payee share the same bank — are handled internally by that bank and never enter NPCI clearing.
DrawerPayeePresentee Bank
2
Presentee Bank Scans, Validates and Uploads via SFTP
The presentee bank scans the physical cheque (CTS-2010 image standards), extracts the MICR line, performs basic due diligence (signature spot-check, account validity, CTS-2010 image compliance), bundles instruments into a lot, and uploads a batch file to NPCI's SFTP server. No per-instrument confirmation is received — only batch-level SFTP transfer acknowledgement.
Presentee BankNPCI SFTP
3
NPCI Accumulates Batches from All Presentee Banks
NPCI's NGCH collects batch files from all presentee banks across all three clearing grids (Mumbai, Delhi, Chennai). Instruments from hundreds of banks are accumulated over the clearing session window. No real-time processing — NPCI holds until the batch cycle triggers.
NPCI / NGCH
4
NPCI Sorts and Routes to Drawee Bank Buckets
NPCI processes the accumulated batch, validates instrument images against CTS-2010 standards, extracts MICR data, sorts instruments by drawee bank IFSC, and places the sorted batch files into each drawee bank's designated SFTP bucket. The drawee bank has no notification — it must poll to discover new items.
NPCI / NGCH
5
Drawee Bank Polls and Fetches Its Inward Instruments
The drawee bank polls its SFTP bucket at a configured interval (typically every 5–20 minutes). When new files are detected, it downloads the batch. This polling gap is the primary source of wasted IET headroom — the bank's 3-hour clock started when NPCI received the item, but the bank is unaware for the first 5–20 minutes.
Drawee BankNPCI SFTP
6
Drawee Bank Performs Due Diligence and Files Return or Confirms
The drawee bank processes each inward instrument: verifies signature, checks account balance, validates against stop payment instructions, screens for fraud, and either confirms (approves payment) or returns (rejects with a return reason). Returns are filed to NPCI's SFTP server. Silence within the IET window is treated as confirmation — the bank pays regardless. This is the critical IET risk moment.
Drawee BankNPCI SFTP (Return File)
7
NPCI Performs Net Settlement Across All Banks
At session close, NPCI calculates the net position for each bank across all instruments — confirmed items create credit (presentee bank) and debit (drawee bank) positions. NPCI submits net settlement instructions to RBI, which executes the inter-bank transfers through the settlement accounts held at RBI's e-Kuber system.
NPCI / NGCHRBI / e-Kuber
8
NPCI Places Settlement Files in Bank Buckets
NPCI generates settlement reports — confirmed instruments, return instruments, net settlement position per bank — and places them in each bank's SFTP settlement bucket. Again, no push notification — banks must poll to discover settlement files are available.
NPCI / NGCH
9
Banks Fetch Settlement Files
Both presentee and drawee banks poll their SFTP buckets, fetch the settlement files, and validate the contents against their submitted/processed instrument records. Any discrepancy between submitted instruments and settled instruments triggers a reconciliation exception that must be resolved manually.
Presentee BankDrawee BankNPCI SFTP
10
Banks Create Counter Entries in Their Core Banking System (CBS)
Based on the settled position, each bank creates the corresponding accounting entries in its CBS: credit to payee's account (presentee bank) and debit from drawer's account (drawee bank) for confirmed instruments; reversal entries for returned instruments. The clearing cycle is complete.
Presentee Bank CBSDrawee Bank CBS

Same-Bank Exception (Important): When drawer and payee hold accounts at the same bank, the cheque never enters NPCI clearing. The bank processes it internally — debit drawer, credit payee — without any SFTP interaction. This volume can be significant for large banks (HDFC, SBI process crores of same-bank instruments monthly) and represents an immediate AI intelligence opportunity independent of NPCI timelines.

The Imperative for Change — Six Structural Failures
Why SFTP Cannot Serve a Real-Time IET Mandate
Failure 01

Silent Inward Item Arrival

Drawee banks discover inward items only at their next SFTP poll — 5 to 20 minutes after NPCI receives them. In a 3-hour IET window, this silent latency is structurally incompatible with safe clearing. A bank processing 500 cheques burns 11% of its IET window before processing even begins.

↳ Up to 11% of IET window wasted on polling delay alone
Failure 02

Zero Per-Instrument Error Feedback

Batch-level rejection provides no instrument-level detail. A presentee bank submitting 500 instruments discovers only that "the batch failed." Identifying which instruments failed, for what reason, and the remediation path requires manual human investigation across SFTP log files.

↳ Manual triage consuming critical operations bandwidth
Failure 03

No Machine-Readable Status API

Live status of a submitted instrument is unavailable during the clearing window. An AI agent processing system has no way to query NPCI about a specific instrument's current state — whether it has been received, validated, routed, or is at IET risk. Status arrives only at session close in a settlement file.

↳ No IET risk visibility until it is too late to act
Failure 04

No Native AI Integration Surface

Modern clearing requires AI agents for fraud detection, signature verification, alteration detection, and CCTV correlation. SFTP provides no structured API surface for these agents. Each bank builds its own file-parsing integration, introducing inconsistency, latency gaps, and unverifiable processing quality.

↳ Each bank's AI integration is a custom, unverifiable point solution
Failure 05

Drawer Has No Digital Identity in the Process

The person who wrote the cheque — the drawer — has zero digital presence in the current CTS process. They cannot track their cheque's status, cannot receive notifications about processing outcomes, and cannot participate in dispute resolution. This is a decade behind UPI's real-time payment experience.

↳ Drawer experience is 1970s in a 2024 payments landscape
Failure 06

Settlement Information Arrives Too Late

Settlement files are available only at session close — hours after instruments were processed. Banks cannot update CBS in real time, cannot notify customers of cheque outcomes, and cannot identify settlement discrepancies until after the window closes. Reconciliation exceptions are discovered late and resolved slowly.

↳ Customer experience is opaque; reconciliation is reactive not proactive

Vision 2030 — The Four-Phase Modernisation Roadmap
Progressive · Non-Disruptive · Backwards-Compatible at Every Step

Master Design Principle: Each phase delivers standalone value. No bank is forced to migrate before its readiness. SFTP and API operate in parallel for minimum 12 months at each transition. Phase 1 requires zero NPCI infrastructure change — banks can deploy today. The IET mandate creates urgency for Phase 1; the intelligence economy and customer experience drive Phases 3 and 4.

Phase 1 Go Live: January 2027
Smarter Bank — Same NPCI Pipes
AI + Parallel Agents + Temporal Workflows at Bank End
Keep SFTP, keep NPCI's file formats, change nothing at NPCI. Inject Vision AI, parallel processing agents, and long-running workflows inside the bank. Collapse internal processing from hours to sub-minute. Achieve 30-minute clearing cycle without asking NPCI to change a single byte. Simultaneously address IET mandate compliance for all 1,581 member banks.
Local Vision LLM Parallel Agents Temporal Workflows Same SFTP <600ms Decision
→ IET breach rate: 0.000% · 30-min clearing cycle · Zero NPCI dependency
Phase 2 Go Live: September 2027
NPCI Gets an API — Near Real-Time
REST API + Webhook Push · Settlement at 30-Min Intervals
NPCI exposes REST APIs. Presentee banks push one instrument at a time (or configurable micro-batches). NPCI webhooks the drawee bank within seconds of receipt. Drawee bank responds via API. Settlement rhythm stays at 30-minute batches — but every individual instrument is tracked in real time. Full IET window available from moment of NPCI receipt.
POST /cts/v1/instruments Webhook Delivery GET /cts/v1/status 30-min Settlement
→ Full IET window from NPCI receipt · Per-instrument tracking · Zero SFTP polling
Phase 3 Go Live: December 2027
MCP-Native Clearing Intelligence
Model Context Protocol · Socket Transport · Resource Streaming
Same logic as Phase 2 — same 30-minute settlement, same actors — but the transport layer graduates to MCP (Model Context Protocol). Instruments flow over persistent sockets. Cheque images and settlement files become MCP Resources pushed and consumed natively. Banks and NPCI speak AI-agent protocol. Clearing grid becomes a native capability in LLM-driven workflows.
MCP Tools MCP Resources Socket Transport AI-Native
→ Clearing grid as AI agent tool · Document streaming as MCP resource · Maximum composability
Phase 4 Go Live: September 2028
Mobile-First Clearing — UPI-Like Experience
Drawer Mobile App · Lien Mark · Real-Time Notification
The drawer uploads a scan of the cheque via their bank's mobile app and fills in beneficiary details. NPCI routes to the drawee bank in real time. Drawee bank confirms and lien-marks the amount immediately. NPCI routes confirmation to the beneficiary's bank. Settlement still executes at 30-minute intervals. Drawer and beneficiary receive real-time status — like UPI, but for cheques.
Mobile App Lien Mark Real-Time Notify NI Act Sandbox
→ Cheque experience at UPI speed · Drawer digital identity · Full lifecycle transparency

Design Principles — Governing Every Architectural Decision
Eight Non-Negotiable Principles Across All Four Phases
01
Backwards Compatibility at Every Phase Transition
SFTP and the new protocol run simultaneously for a minimum of 12 months at each phase boundary. No bank is forced off SFTP before it is ready. Dual-mode operation is a first-class requirement, not an afterthought. A bank on Phase 1 receives full IET safety guarantees while operating purely on SFTP.
02
Instrument-First, Not Batch-First
Every API operation, every MCP tool call, every mobile interaction addresses a single instrument with its own state machine, IET timer, error record, and audit trail. Batching is a transport optimisation — not a protocol concept. Each instrument is independently trackable, queryable, and auditable at any moment.
03
Authentication Before Data — Three Layers Minimum
Every request carries: (L1) institutional mTLS certificate from NPCI-issued PKI, (L2) API Key + session token (30-minute TTL) from NPCI auth server, (L3) HMAC-SHA256 request signature over canonical string. No instrument data is processed unless all three layers pass independently. Applies from Phase 2 onwards; Phase 4 adds mobile device attestation as L4.
04
Idempotency as a First-Class Guarantee
Every state-changing operation requires an X-Idempotency-Key (UUIDv7) with 24-hour NPCI-side deduplication. A retry with the same key returns the original response without reprocessing. This eliminates duplicate NGCH submissions under any network retry condition — permanently. A duplicate filing is never possible if the key policy is followed.
05
Machine-Readable Errors at Every Layer
Every error carries: numeric error code, category (AUTH/INSTR/SYS/IMG/RATE/SETTLE), human-readable description, remediation hint, affected field path, and correlation ID. No error is a black box. A bank's AI processing system must classify any error and route it to the correct remediation without human intervention — including the emergency IET path.
06
IET Safety Is Structural, Not Configurable
The IET watchdog fires at T-30 seconds regardless of any configuration, any API mode, any network condition. This is hardwired in the bank-side processing architecture (ASTRA). An emergency filing path — to SFTP if REST is unavailable, to REST if webhook is unavailable — is always active. IET breach rate target: 0.000% across all phases.
07
Data Localisation — No Instrument Data Leaves India
All cheque images, MICR data, customer information, and processing decisions remain within the bank's on-premises infrastructure and NPCI's India-based grid. No cloud processing, no cross-border data flows, no third-party AI API calls with customer data. This applies to all four phases including Phase 4 mobile capture. LLMs run on-premises at the bank.
08
Synchronous Acknowledgement, Async Processing
Submission APIs return synchronously within 200ms SLA with a receipt confirmation. The instrument then undergoes asynchronous NPCI validation and forwarding. Status is queryable at any time. Outcomes are delivered via webhook/MCP push — no polling required from Phase 2 onwards. The 200ms synchronous window is the only moment where latency is a bank-side obligation.

Phase 1 Deep-Dive — AI-Powered Bank-Side Processing
Go Live: January 2027 · Zero NPCI Dependency · Full IET Compliance

Phase 1 Principle: NPCI does not need to change a single byte. Banks deploy ASTRA — the AI clearing intelligence platform — and immediately gain IET safety, fraud detection, and 30-minute clearing cycles. This phase is fundable and deployable today, independent of NPCI's modernisation timeline.

Outward Clearing
Presentee Bank Role — From Scanner to SFTP Submission
SFTP Unchanged
Physical Scanner → CTS-2010 Image

High-resolution scanner captures front and back. MICR line extracted via GOT-OCR2.0 (99%+ accuracy). CTS-2010 compliance validated automatically.

Vision AI — Alteration Detection

Qwen2-VL 72B analyses image for overwriting, erasure, correction fluid, ink differences. Tamper risk score computed per field — amount, date, payee, signature area.

Lot Creation and Endorsement

Instruments batched into lots, endorsement stamp applied digitally, lot sealed and flagged for SFTP submission. Sub-member bank (UCB) instruments routed through sponsor bank.

SFTP Submission + Session Reconciliation

Lot submitted to NPCI SFTP. Return Reason File generated at session close. Settlement file fetched and reconciled automatically — exceptions flagged to ops workstation.

Inward Clearing — IET Critical Path
Drawee Bank Role — 500 Parallel Agents in <600ms Wall Clock
IET Watchdog Active

Each inward cheque spawns one dedicated AI agent. 500 cheques → 500 parallel agents → entire batch decided in <600ms wall clock. The IET watchdog runs as a sibling process for every instrument — if processing stalls, it files directly to NPCI at T-30 seconds before IET expiry.

1. OCR + Alteration Detection

GOT-OCR2.0 extracts all fields. Qwen2-VL detects alterations. If confidence < threshold → human review, not auto-return.

2. Signature Verification

Siamese neural network compares against CBS specimen vault. Vault miss → human review. Match score < threshold → human review. Never auto-return on vault miss.

3. Positive Pay Check

PPS vault lookup for banks with Positive Pay mandate. Mismatch or vault miss → human review. PPS confirmation adds to fraud score confidence.

4. CBS Balance + Stop Payment

Real-time CBS query for balance adequacy and stop payment instructions. CBS unavailable → image-only path, human review — never auto-return.

5. Fraud Scoring + SHAP

XGBoost ensemble + Llama 3.3 70B synthesis. SHAP values computed for every decision. Score > threshold → human review. Score stored before NGCH filing.

6. STP Decision + NGCH Filing

Score < STP threshold + all checks pass → auto-confirm filed to NGCH. Score > threshold → human review queue. IET watchdog monitors all paths.

IET Safety
IET Watchdog Workflow — T-30 Second Emergency Filer
Non-Negotiable

For every inward instrument, a sibling IETWatchdogWorkflow is spawned as the very first action — before any AI activity begins. It monitors the remaining IET window. At T-30 seconds before IET expiry, if no decision has been filed, it immediately files an emergency confirmation directly to NGCH. This fires regardless of the main workflow state — network conditions, AI model availability, or human reviewer delays.

Spawn Order

IETWatchdogWorkflow is always the first child workflow. No activity executes before the watchdog is active and confirmed started.

Parent-Close Policy: ABANDON

Watchdog survives parent workflow failure. If the main processing workflow crashes, the watchdog remains active and will still file at T-30s.

Emergency Path

SFTP direct submission as fallback if REST/MCP unavailable. The emergency path is always tested in smoke tests — it cannot be disabled.

Audit

Every watchdog firing is written to Immudb as a CRITICAL audit event. Emergency filings are surfaced on the ops workstation in real time.
