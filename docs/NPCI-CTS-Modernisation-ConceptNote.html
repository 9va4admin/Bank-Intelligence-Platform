Silence = Confirmed

The Structural Flaw: A drawee bank learns about an inward item only at its next SFTP poll cycle — typically 5 to 20 minutes after NPCI receives it. In a 3-hour IET window, this silent latency is unacceptable. A bank processing 1,000 cheques in the AM session is consuming 10–20 minutes of its IET headroom before it even knows the items have arrived.

2
The Imperative for Change — Five Structural Failures
Why SFTP Cannot Serve a Real-Time Clearing Mandate
Failure 01

No Real-Time Receipt Confirmation

When a presentee bank submits a batch via SFTP, confirmation arrives only in the next polling cycle — 5 to 15 minutes later. A failed submission may not be discovered until IET headroom has already been consumed.

↳ IET exposure from submission uncertainty
Failure 02

Zero Per-Instrument Error Feedback

Batch-level rejection provides no instrument-level detail. A bank submitting 500 instruments in a batch discovers only that "the batch failed." Identifying which instruments failed, why, and what to correct requires manual investigation.

↳ Manual triage consuming operations bandwidth
Failure 03

Polling Creates Artificial Inward Latency

Inward items arrive at NPCI in real time. The drawee bank discovers them only at its next poll cycle. This introduces 5–20 minutes of artificial latency before processing can begin — directly consuming IET headroom that belongs to the bank.

↳ Up to 11% of IET window wasted on polling delay
Failure 04

No Machine-Readable Status API

Live status of a submitted instrument is unavailable during the clearing window. Settlement reports arrive only at session close. A bank cannot determine in real time whether an item has been delivered, is being processed, or is at IET risk.

↳ No IET risk visibility until it is too late
Failure 05

No Native AI Integration Path

Modern processing systems use AI agents for fraud detection, signature verification, and decision-making. SFTP provides no structured API surface for these agents — every bank builds its own translation layer, introducing inconsistency, latency, and unverifiable processing gaps.

↳ Each bank's AI integration is a custom point solution

Progressive, Non-Disruptive, Backwards-Compatible

Design Principle: Each phase builds on the previous. No bank is forced to migrate before its readiness. SFTP and API operate in parallel for 24 months. The IET mandate creates urgency for Phase 1; the intelligence economy drives Phase 3.

Layer
Months 0–12
Replace SFTP file exchange with synchronous REST APIs. Individual instrument submission, real-time callback delivery, structured per-instrument error codes, and live status queries — at UPI-standard quality. Clearing mechanics unchanged.

API Design Principles — The Foundation
Five Principles Governing Every Design Decision
01
Instrument-First, Not Batch-First
Every API operation addresses a single instrument. Batching is handled at transport layer (bulk array), not at protocol layer. Each instrument gets its own response, error feedback, lifecycle state, and IET timer — independent of every other instrument in the submission.
02
Authentication Before Data — Always
Every request carries two identity layers: institutional identity (mTLS certificate + API key) and request integrity (HMAC-SHA256 signature over the payload body). Data is never processed unless both layers pass. This mirrors UPI's device authentication → transaction authorisation model.
03
Synchronous Acknowledgement, Async Processing
Submission APIs return synchronously with a receipt confirmation within 200ms SLA. The instrument then undergoes asynchronous NPCI validation and forwarding. Status is queryable at any time. Outcomes are delivered via webhook callbacks — no polling required.
04
Machine-Readable Errors at Every Layer
Every error carries: numeric error code, category code, human-readable description, remediation hint, affected field path, and correlation ID. No error is a black box. A bank's processing system must be able to classify any error and route it to the correct remediation path without human intervention.
05
Idempotency as a First-Class Guarantee
Every state-changing operation requires an X-Idempotency-Key header (UUIDv7). NPCI stores idempotency keys for 24 hours. A retry with the same key returns the original response without reprocessing. Eliminates the possibility of duplicate instrument submissions under network retry conditions — permanently.
06
IET-Aware at Protocol Level
Every instrument response and status query includes iet_deadline, iet_remaining_seconds, and iet_risk_level. The protocol is not neutral to time — it actively communicates urgency. Banks' processing systems receive continuous IET state without needing to calculate it independently.
5
Authentication Architecture — Three Mandatory Layers
Zero Trust · mTLS · HMAC-SHA256 · Token Scoping

Architecture Principle: No layer can be skipped. Failure at Layer 1 (mTLS) means no TCP connection — no HTTP response is ever issued. Failure at Layer 2 (API key + session token) means a 401 before any payload is parsed. Failure at Layer 3 (request signature) means a 401 even if credentials are valid — the request integrity must be independently verified.

NPCI → Drawee Bank: Real-Time Inward Push (Webhook)
POST {bank_registered_webhook_url} · 5-Second Response SLA
Upon accepting an instrument, NPCI immediately pushes it to the drawee sponsor bank's registered HTTPS webhook endpoint. This is the core change from SFTP polling — the drawee bank's IET clock starts from NPCI receipt, not from the bank's next poll cycle. NPCI retries 3 times on non-2xx responses (30s, then 120s).
Headers Sent by NPCI to the Bank's Webhook
X-NPCI-Delivery-ID:    del_01J4K7M9P2Q3R4S5T6U7V8W9X0     ← Unique per delivery attempt
X-NPCI-Signature:      {hmac_sha256_using_shared_webhook_secret}
X-NPCI-Timestamp:      {epoch_ms}
X-NPCI-Retry-Count:    0                                   ← Increments: 0, 1, 2
Content-Type:          application/json
X-API-Version:         1.0

Drawee Bank → NPCI: Return Filing
POST /cts/v1/returns · Images NOT Re-transmitted · Metadata Only
When the drawee bank decides to return an instrument, it files a structured return via API. Images are NOT re-transmitted — NPCI holds the originals. Only metadata and the return reason code are sent. IET status is returned synchronously, confirming the return was filed within the window.

Phase 3 — MCP Intelligence Layer
AI-Native Clearing · From REST Integration to Agent-Native Tool

The Strategic Shift: In Phase 1, AI systems at member banks call an API. In Phase 3, AI agents use the clearing grid as a native tool within their reasoning workflow. The difference is not cosmetic — it is the difference between a bank that has integrated clearing and a bank where clearing is a first-class capability of its intelligence infrastructure. India pioneered UPI. India can pioneer MCP-native clearing.

🔧MCP Tools — Actions the Agent Can Invoke
submit_instrument
Submit scanned instrument. Returns npci_reference, iet_deadline, delivery_status synchronously.
file_return
File return for inward instrument. Returns return_reference, iet_status (FILED_WITHIN_IET / IET_BREACH_RISK).
query_instrument_status
Real-time status, IET countdown, lifecycle events for any instrument by npci_reference.
query_duplicate_history
Check prior presentations of a cheque (MICR + account hash). Returns count, dates, outcomes. No PII in response.
query_session_state
Current clearing session status, IET deadline, pending instrument count, risk level for a given zone.
get_return_reason_codes
Machine-readable return reason code definitions with applicability rules. Agent uses to select correct return code.
📦MCP Resources — Structured Data the Agent Can Read
npci://clearing/{zone}/session/current
Live session state, IET deadline, submission cutoff. Updated in real time by NPCI. Agent reads before processing decisions.
npci://clearing/{zone}/risk/signals
Aggregate cross-bank fraud signals for the current session. Example: "Code 11 returns up 340% in MUMBAI zone today." No per-bank or per-instrument data.
npci://member/{bank_ifsc}/position/current
Bank's own net clearing position in the current session. Own bank only — cross-bank position data is never exposed.
npci://system/health
NPCI infrastructure health per clearing zone. Agents use to trigger degradation handling before encountering errors.
npci://instruments/{npci_ref}/context
Enriched instrument context: presenting bank info, NPCI validation flags, IET status, clearing session assignment.

Migration Roadmap — Non-Disruptive, Bank-Controlled
24-Month Parallel Operation · 30-Month SFTP Notice · No Forced Cutover
Phase 1 · REST API
Months 0–12
Months 0–3: NPCI API infrastructure ready. Sandbox live. Mandatory UAT environment available to all 1,581 members.
Months 3–6: Pilot with 10 Sponsor Banks. Full flow testing: submission, webhook delivery, return filing, status query.
Months 6–9: Open onboarding for all Sponsor Banks. SFTP continues in parallel — no bank disrupted.
Months 9–12: Measure adoption. Publish API vs SFTP latency comparison. IET incident rate tracking by protocol.
Phase 2 · Webhook Push
Months 12–24
Months 12–15: Webhook push infrastructure deployed. Banks register endpoints via member portal. Pilot with 5 high-volume SBs.
Months 15–18: Open webhook registration. Banks choosing webhooks get immediate full-IET-window benefit.
Months 18–21: Measure: banks on webhook show <5s inward latency vs 5–20min SFTP. Publish data to drive adoption.
Month 24: SFTP sunset notice formally issued. 30-month countdown begins. No bank loses SFTP access until Month 54.
Phase 3 · MCP Layer
Months 24–36
Months 24–27: MCP server specification published. Tool catalog and resource schema finalized with industry input.
Months 27–30: MCP sandbox available. Banks with AI-native processing systems begin integration testing.
Months 30–33: MCP production go-live. AI agents at member banks use NPCI as a native clearing tool.
Month 36: Cross-bank fraud signal sharing via MCP resource layer. India's clearing grid becomes an intelligence network.
Bank Onboarding Requirements

Step 1: Webhook URL registration. Step 2: NPCI PKI certificate issuance. Step 3: API key provisioning. Step 4: Mandatory 5-step UAT completion. Step 5: Production go-live with parallel SFTP.
